00
Core Objective
What PromptLog is — and what it is not
Problem Statement

Your backend already generates system prompt iterations, transcript outcomes, and improvement suggestions. The problem is none of this is visible, comparable, or shareable in a structured way. Decisions get made in Slack threads, prompts live in spreadsheets, and stakeholders have no audit trail.

PromptLog is a Documentation + Visualization Dashboard. It consumes data your backend pushes and turns it into a living, auditable record of prompt evolution — organized by project, browseable by anyone, and actionable by the team.

📡
Data Consumer, Not Producer
PromptLog never generates prompts or suggestions. It only displays what your backend pushes via a REST webhook.
🔍
Source of Truth
Every prompt version, transcript, and suggestion is stored, versioned, and searchable — forever.
🤝
Stakeholder-First UX
Non-technical managers and clients can read, comment, and share without needing to understand the ML pipeline.

Data Pipeline — The Push Model
How data flows from your backend into PromptLog

Every time a voice session ends, your backend fires a single HTTP POST to POST /api/ingest. The payload below is the contract.

// Inbound payload — pushed by backend after each voice session { "project_id": "sales_q2_2026", "project_name": "Sales Outbound Bot", "prompt_version": "v1.14.2", "system_prompt": "You are a friendly SDR assistant...", "transcript_id": "tx_0x9f2a", "raw_transcript": [{ "role": "user", "text": "..." }, ...], "llm_outcome": { "status": "success", // "success" | "failure" | "needs_review" "sentiment": "positive", "goal_achieved": true, "latency_ms": 1240, "token_cost": 0.0032 }, "improvement_suggestions": [ { "id": "sug_01", "priority": "high", "text": "Shorten the intro by 2 sentences..." }, { "id": "sug_02", "priority": "medium", "text": "Add a fallback when prospect asks for pricing..." } ], "metadata": { "timestamp": "2026-05-02T09:14:22Z", "session_duration_s": 183, "pushed_by": "backend-worker-03" } }

↳ All fields are required except improvement_suggestions (optional if backend has no output for this session).

03
Functional Requirements
Screen-by-screen breakdown of features and user actions
Screen A
Global Dashboard
All Users

Entry point. Shows a card per project with a live health pulse. Designed to give a 10-second status read without drilling in.

  • Project cards showing name, latest push timestamp, and rolling 7-day success rate badge
  • Create New Project (admin only) — sets up a project silo and generates the webhook URL
  • Global notification bell with alerts for drift, new versions, and @mentions
  • Quick-switch between project silos via top navigation rail
Screen B
Evolution Timeline
Designer / Eng

The core screen. A vertical, git-style timeline of every prompt version pushed by the backend. The audit trail that answers "why did the bot change?"

  • Vertical chronological timeline; each node shows version number, push timestamp, and outcome delta vs previous version
  • Click any node → Snapshot View: full prompt text in a code block + a sample of the transcripts from that version
  • Select two nodes → triggers Side-by-Side Diff View (red/green line highlights, no external tools needed)
  • Add Manual Note on any version (e.g. "Deployed for Summer Campaign") — visible to all team members
  • Pin a version as "Stable Baseline" — highlighted with a gold indicator on the timeline

Screen C
Transcript Viewer
Designer / Eng

Three-panel layout: transcript list rail on the left, chat-style transcript in the center, and the backend Insight Panel on the right.

  • Left Rail: List of sessions with Duration, Sentiment chip, and Success / Failure / Needs Review tag — filterable
  • Center Panel: Clean chat-bubble transcript UI; speaker labels (Bot / User); timestamps per turn
  • Right Insight Panel: Backend improvement suggestions listed by priority; outcome metadata (latency, token cost)
  • "Promote to Wiki" button on any suggestion — saves it to the project Knowledge Base with one click
  • "Share" button — generates a shareable deep link to this specific transcript (view-only for recipients)
Screen D
Knowledge Base
All Users

The institutional memory layer. Captures what worked across all projects so insights from Sales don't die before Marketing can use them.

  • Per-project "Learnings" section populated by promoted suggestions
  • Global Rulebook: a curated list of system prompt snippets marked as "Company Standard" — enforced across all projects
  • Categorize entries with tags: Tone of Voice, Closing Techniques, Objection Handling, etc.
  • Full-text search across all projects and all categories
  • Export any rulebook snippet as a formatted snippet for direct paste into your dev environment

Key Workflows
End-to-end user journeys through the platform
Workflow 1 — The Review Loop Designer Path
1
Notification: "New optimization suggestions available for Sales Project." Push notification or in-app bell.
2
Entry: Open Transcript Gallery. Filter by "Needs Review." Find the flagged session.
3
Analysis: Read the transcript. Review backend suggestions in the Insight Panel.
4
Decision: If the suggestion is valid, update the system prompt in your dev environment. Backend pushes the new version.
5
Promote: Click "Promote to Wiki" on the key suggestion for future reference.
6
Closing: Timeline auto-updates with the new version node. Add a manual note.
Workflow 2 — Stakeholder Handoff Manager Path
1
Trigger: Manager notices the bot sounds different on calls and asks why.
2
Lookup: Opens the Evolution Timeline for the Sales project.
3
Compare: Selects "Current Version" vs "2 versions ago." Diff View loads instantly.
4
Context: Reads the designer's manual note explaining the change rationale.
5
Share: Copies a shareable link to that diff view and sends it to the client for sign-off.

UX Priorities & Design Constraints
Non-negotiable design decisions for stakeholder-friendly high-density UI
Component Design Goal Constraint
Prompt Card System prompt text in a monospace code block with syntax-level line numbering Max visible height ~300px with scroll; never truncate without affordance
Diff View Side-by-side default; toggle to inline for long prompts Red = removed lines, Green = added lines; no color beyond these two for diffs
Transcript Panel Chat-bubble UI; Bot vs User visually distinct; timestamps per turn Panel must be readable on 1280px screens alongside the Insight Panel
Insight Panel Suggestions ordered by backend priority (high → low); each has a "Promote" action Must not obscure or compete visually with the transcript
Empty State When no transcripts pushed: show an onboarding card with the webhook URL and integration guide Empty state must be actionable, not just decorative
Shareable Links Deep links to: specific transcript, specific version, or specific diff view Recipients without accounts get a read-only token-based view (no login required)
Search Global full-text search across all transcripts, prompt text, and wiki entries Results must show context (project, version, date) — not just raw hits

07
Notifications & Triggers
Automated alerts that keep the team informed without manual polling
System Prompt Drift Alert
Triggered when the backend detects ≥3 consecutive "failure" outcomes on the same prompt version. Flags the project for urgent review.
New Version Pushed
Fires whenever the backend pushes a new prompt_version. Notifies assigned team members with a link directly to the new timeline node.
Optimization Suggestions Available
Sent when a session push includes non-empty improvement_suggestions. Links directly to the Transcript + Insight Panel for that session.
Knowledge Milestone
Celebratory trigger when a project hits its 10th version or 100th transcript — surfaced as a banner in the Timeline View.
Manual @Mention
Any team member can @mention a colleague in a timeline note or transcript comment. Delivers an in-app and email notification with direct link.

Technical Stack
Recommended architecture for a lean, maintainable internal tool
Frontend
Next.js 14
App Router; SSR for shareable links; React Server Components for performance
Styling
Tailwind CSS
Utility-first; consistent spacing; dark theme tokens baked in
Database
PostgreSQL
Relational model handles Projects → Versions → Transcripts → Suggestions naturally
Diff Engine
diff-match-patch
Google's open-source library for character-level text diffing; battle-tested
Ingest API
REST Webhook
POST /api/ingest — authenticated via API key in header; idempotent by transcript_id
Auth
NextAuth.js
SSO / Google Workspace; role-based; shareable link tokens via JWT
Search
pg_trgm
PostgreSQL trigram full-text search — no extra service needed for V1
Hosting
Vercel + Supabase
Zero-infra setup for an internal tool; scales easily; Postgres managed by Supabase
09
Out of Scope — V1
Explicitly excluded to keep scope tight and ship fast
AI / ML Within PromptLog
PromptLog does not generate, score, or modify prompts. All ML logic lives in your backend. PromptLog only displays what is pushed.
Prompt Editor / IDE
No in-browser editing of system prompts. All prompt changes happen in your dev environment and are pushed via the backend webhook.
Voice Playback
V1 handles text transcripts only. Audio playback of voice sessions is deferred to V2.
Custom Analytics / BI
Charts showing success rate trends are in scope. Custom report builders, export to BI tools, and advanced funnels are V2.
